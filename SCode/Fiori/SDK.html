SAP Fiori for iOS

6 Things I Learned From Designing SDK Components

Mar 16, 2019

What is an SDK?

While our team is working on the design system, our other main task is to support the SDK team to develop reusable components. Designing for SDK components adds another layer of complexity onto the already challenging enterprise scenarios. To provide some context, an SDK (Software Development Kit) is a set of development resource for engineers to build products efficiently. The SAP Cloud Platform SDK for iOS is built on top of the iOS UIKit, which means product teams will be able to utilize these two frameworks simultaneously. It’s designed to be flexible enough to support use cases across a wide range of industries such as project management, finance, clienteling, production management, sales, health care and more.

Challenges

The biggest challenge of designing SDK components for me was to create specifications that would work well for every variant of the component I was working on. I worked closely with developers to understand technical constraints, as well as defining details such as whether a text label would truncate or not. I realized it’s a matter of balancing flexibility and contraints in each component to ensure product teams can use it within the best practices we defined, and are able to customize the component without a lot of extra effort. Despite of the challenges, I found myself picking up a few critical concepts that were not taught in school and I would like to share them here:

1

Modularity—Provide flexibility by desinging a modular structure

The object header was the first component I ever designed and I learned so much from the process. I started by asking the question: what does a header accomplish? By collecting relevant app examples, I found that object screens usually display the most important or most frequently used information at the top most section, allowing users to grasp critical information quickly.

The next question I had to tackle was how can this component be flexible enough to fit a variety of use cases? I looked into the app examples by identifying what type of content is usually included, and designed a structure based on modular blocks. The image below showcases how flexible the object header can be. The only manatory piece in this component is the main info block, the rest of the pieces are designed to move based on screen size and availability of other blocks.

The possibility of combination of modules is endless. Due to the versatility of this component, it has been widely used in different apps supporting a variety of scenarios.

2

Priority—Providing constraints by setting max values for lower prioritized content

Providing constraints is as important as providing flexibility when designing components. After designing the modular blocks of the object header, the problem was to define which block has the highest priority in terms of what happens when content in a certain block is lengthy.

It’s common to believe that the main info block has the highest priority, however I decided to prioritize the content blocks from right to left after many rounds of exploration. Since the status block would most likely contain the least content, and it is critical that it has enough space to display messages that require the users’ attention, it should take first priority in the component as well as a maximum width. The maximum width has been tested to fit at least two full words, and if the phrase exceeds the limit, it can wrap to multiple lines without being unreadable. The middle section with second priority usually contains a chart or a paragraph. The maximum width has been defined by the maximum width of the chart. Lastly, the left-most section utilizes the remaining space in the component but is still able to display content in a readable width.

3

Responsiveness—design a different layout for different screen sizes

We have been continuously testing our components, making sure that they function properly across all device sizes and device orientation. Sometimes a component requires a completly new layout or treatment for it to adapt to a drastically different screen size. After receiving feedback from a few product teams, we decided to enhance our text fields to make them more flexible.

The text field implemented initially has a stacked layout, with the key label on top and the value label stacked below it. In compact width(iPhone), the stacked layout works fine as both labels have plenty of room to grow. While in regular width(iPad), the stacked layout could become uncessary as horizontal space is abundant. Switching the text field to a single-line layout in regular width saves valuable vertical space especially when the input value is short. The layout of the text field is now configurable in the SDK to accomodate more use cases.

Accessibility—make sure content has plenty of space to grow

Accessibility has been an intriguing topic for me to discover as a designer. I understand that accessibility covers a lot of different areas, but here I would like to focus on layout and dynamic type.

The Fiori for iOS design system and the SDK use the standard font in iOS–San Francisco, and support dynamic type, allowing users to adjust text size settings based on their preferences. We as a team investigated into best practices of how componets should adapt to larger font size settings, and enhanced our components by designing layouts specifically for this purpose.

6

Validation—validate interactions with users to avoid designing based on assumptions

Currently I’m working on an OCR business card scanning component with the help of a user researcher. We have been gathering expectations from interviews and testing multiple prototypes. I was inspired by the feedback we received and it changed my perspective on many assuptions I made earlier. I will be sharing more when this component is released.

Some Thoughts

Designing SDK components changed the way I design apps. The reason is because after going through the process of considering every possibility of how a component would perform, I find myself becoming more mindful of how a design element behaves under different situations. It has been a valuable experience for me to learn and grow by working with developers to understand technical concepts. Seeing native apps built with the SDK brings me a great deal of fufillment.
